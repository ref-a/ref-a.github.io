Rutinen for nye brukernære behov

  1. Melde behov, gjerne av datakonsumentene

    1. Interessenter melder ønsker inn gjennom kjent verktøy / grensenitt

    2. Periodisk prosess for å samle innspill og prioritere

    3. For de case som går videre: Intervjue interessenter/ kjøre tjenestedesign workshop for å beskrive scenariet

  2. Analysere behov og finne ut hvilke domener og kilder som er involvert

    1. Hvem gjør dette? Tjenesteansvarlige? Informasjonsdomeneeiere? Prosesseiere? Sekretariatet? Hvem hos institusjonene?

    2. F. eks. studieadministrasjon kunne være en domene, og tjenesansvarlig for FSAPI

  3. Finn eksisterende datasett involvert (helst fra datakatalogen) som er ferdig harmonisert

    1. Spesifisere hva som blir berørt f. eks. begreper, API, datamodell, (tjeneste- og API-eiere)

  4. Identifiser hva må skapes, harmoniseres eller videreutvikles i tillegg til eksisterende datasett for å tilfredsstille behovet

    1. Trenger vi nye: begreper, kilder, API, datasett m.m?

    2. Kontakte interessentene til de berørte områdene med sikte på å være med i prosessen

  5. Kjør begreps harmoniseringsprosessen etter behov med relevante interessenter (hele verdikjeden for data involvert må være representert)

    1. Følg standarden om Begrepsharmonisering og begrepsdifferensiering (kurs om begrepsarbeid er en utfyllende ressurs)

  6. Definere API/datasett team med representasjon fra både bruksmiljø og produsent (er dette den samme gruppen fra punkt 5?)

    1. Det kan ta noen iterasjoner med utprøving av API før man er fornøyd

    2. Senere oppdateringer kan kjøres som iterasjoner da man allerede har resultatene fra tidligere steg.
